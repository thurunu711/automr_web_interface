Welcome to

Test Autonomous Driving Models with Confidence

AutoMR is an intelligent metamorphic testing platform designed to evaluate regression-based autonomous driving models; even when the correct output is unknown.

Instead of relying on traditional test cases with predefined expected results, AutoMR automatically generates follow-up tests using carefully designed Metamorphic Relations (MRs) to uncover hidden defects, improve model reliability, and support safer Autonomous Driving systems.

Definition & example

We test a system using only its inputs and outputs. We never look inside. This platform just feeds the model frames and watches what comes out.

Metamorphic Relations

Add extra cheese to a pizza. You don't know the exact taste score. But it should never taste less cheesy.

Definition & example

This is a rule for how output should change when input changes slightly.For example, a small brightness change in an autonomous vehicle camera image should not cause the car to make an extreme steering decision. We ask if the result is consistent, not if it's correct.

04
Robustness Testing
🚗

A driver trained on sunny roads suddenly hits rain and fog. Can they still drive safely?

Definition & example

We check if an AI keeps working when inputs get noisy or blurry. A tiny change in the input should never cause a wild change in the output.

Definition & example

We check that predictions stay logical over time. One frame's result shouldn't wildly contradict the next, unless the scene really changed.

Metamorphic Testing Overview

Metamorphic testing checks relationships between outputs instead of validating exact outputs against fixed labels.

Transformation — T

The recipe that turns the original input into a new test input. It only touches x — it never looks at the model. Example: "rotate the image by 15°."

Relation — MR

The rule that says how the two outputs, y and y′, ought to compare once T has been applied. Example: "rotating the image shouldn't change the predicted class."

They're two different jobs on the same test case: T builds the new input, MR judges the pair of outputs it produces. "Shouldn't change" is only one kind of MR — see the Types of MR tab for the others, and Worked Example for a full run-through with real numbers.
Traditional testing
f(x) = y

Requires a known, exact expected output for every input.

Only requires that the relationship between two outputs holds.

The MR, in plain English

"Lighting shouldn't meaningfully change the steering decision." Nothing here says what the angle should be — only that it shouldn't move much between x and T(x).

The MR, formally
|f(x) − f(T(x))| < ε
|12.4 − 13.1| = 0.7°, checked against ε below
ε = 0.05 × (max(Y) − min(Y)) = 0.05 × (35° − (−5°)) = 2.0°
|Δ| = 0.7° < ε = 2.0° →
PASS
If instead T had been "paint a stop sign onto the road" and the angle barely moved, that same MR would FAIL — the relation would flag a real robustness gap.

"Output shouldn't change" is just the strictest MR — invariance. The real definition is broader: an MR is any rule that says how y and y′ must relate, given how T changed x. Direction, proportion, and bounded drift are equally valid MRs.
These are the exact MetamorphicRelation classes used in this project's test suite — each one checked with check(y1, y2) against a fixed ε.
Global
range-based

One fixed tolerance for the whole dataset, scaled by how spread out the outputs are.

ε = a · (max(Y) − min(Y))
same width for every sample
  • a — sensitivity factor, e.g. 0.05
  • Y — the set of all outputs/labels
Local
prediction-aware

Tolerance adapts to the size of the current prediction, not the dataset as a whole.

|f(x) − f(T(x))| / (|f(x)| + δ) < a
width adapts per sample
  • f(x) — prediction on the original input
  • a — sensitivity factor, e.g. 0.05
  • δ — small constant, e.g. 10⁻⁶, avoids divide-by-zero
Global ε fixes one tolerance for the whole dataset; local ε adapts per-sample, so a prediction near zero isn't held to the same absolute tolerance as a large one.
